# Project Overview

**Strava Running Bot** is a Node.js Discord bot that posts Strava activities from running team members to a Discord channel in real time. It also manages team races (planning, weekly/monthly announcements) and personal bests.

Single-process app: an HTTP webhook server (Strava events + OAuth callback) plus a Discord client running in the same process.

# Architecture

## Flow

1. Member runs `/register` in Discord → bot returns Strava OAuth URL.
2. Member authorizes → Strava redirects to `/auth/strava/callback` → tokens are AES-256 encrypted and persisted via `DatabaseMemberManager`.
3. Strava sends activity events to `POST /webhook/strava`.
4. `ActivityProcessor` fetches the full activity via `strava/api.js` (refreshing tokens as needed), formats it through `EmbedBuilder`, and posts to the configured Discord channel.
5. `Scheduler` (node-cron) fires weekly/monthly race announcements via `RaceManager`.

## Module conventions

- One class per file; filename matches class name (e.g. `MemberManager.js` → `class MemberManager`).
- Managers own state + persistence for a single domain concept (members, races, PBs, settings).
- Utils are stateless helpers.
- Logger is the only allowed I/O wrapper — do not `console.log` in `src/`.
- All Strava token access goes through `strava/api.js`; never call Strava endpoints directly elsewhere.
- All encrypted-at-rest fields go through `utils/EncryptionUtils.js`.

## Database

- SQLite file at `config.database.path` (default `/app/data/bot.db` in Docker, override locally via `DATABASE_PATH`).
- Schema in `src/database/schema.js`; migrations generated by `npm run db:generate`, applied by `npm run db:migrate`.
- `DatabaseManager` is the connection/lifecycle owner; per-domain managers wrap it.

# Deployment

## Local development

```
cp .env.example .env        # fill in DISCORD_*, STRAVA_*, ENCRYPTION_KEY
npm install
npm run db:migrate
npm run dev
```

`PORT` defaults to 3000. Strava webhooks require a public URL — use a tunnel (ngrok, cloudflared) and register the webhook via `utils/setup.js`.

## Docker

```
docker compose up -d        # uses docker-compose.yml
```

Production compose file: `docker-compose.prod.yml`. Image is built and pushed by `.github/workflows/build-and-push.yml`.

# Development Workflow

**MANDATORY** before any change:

1. **Branch check.** Verify you're on a feature branch, not `main`. If not, stop and ask the user to:
   - `git checkout main && git pull`
   - `git switch -c feat/<name>`
2. **Test-first for new features.** Write the test in `__tests__/` mirroring the `src/` path, then implement.
3. **`npm test` must pass — all of it.** Don't relax assertions to make tests pass; fix the code. Cover edge cases.
4. **Coverage + security gate.** New code: ≥ 80 % coverage (90 % preferred), verified via `npm test` (Jest writes `coverage/lcov.info`, uploaded to Codecov by CI). Resolve all CodeQL alerts before merging — zero tolerance for open findings.
5. **Self-review.** Before committing, read the diff critically — would you reject this in code review? Fix what you find, then re-run steps 3–4.

**README architecture sync.** If you add/remove/rename files or directories, update the project-structure tree under `## 🏗️ Project Architecture` in `README.md` and any affected paths in `docs/`.
